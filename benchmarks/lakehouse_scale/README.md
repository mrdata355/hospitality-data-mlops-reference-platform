# Lakehouse Scale Benchmark

This package generates deterministic, synthetic hospitality data with Spark and
records reproducible scale evidence. It is designed to answer a specific
engineering question:

> Can the platform plan, generate, store, validate, recover, and account for a
> large distributed workload without committing customer data or credentials?

It does **not** claim that a 10 TB run has occurred. Only evidence from an
executed run may be described as completed.

## What is included

- `sample`, `100gb`, `1tb`, and `10tb` scale profiles
- 12 operational domains aligned with the platform
- Spark-expression-only generation with no Python UDF bottleneck
- controlled duplicates, nulls, late arrivals, and resort-key skew
- logical size, estimated row, output partition, runtime, file-count, and
  actual-storage evidence
- an explicit cost confirmation gate for every large profile
- an optional full row-count reconciliation pass
- a non-executed evidence template that cannot be mistaken for a real result

## Profiles and progression

| Profile | Logical target | Intended use |
|---|---:|---|
| `sample` | 64 MiB | local or development contract smoke test |
| `100gb` | 100 GiB | integration, partitioning, and cost calibration |
| `1tb` | 1 TiB | distributed scale gate and tuning evidence |
| `10tb` | 10 TiB | final controlled benchmark after the 1 TB gate passes |

The 10 TB plan uses a 256 MiB target file size, which produces approximately
40,960 output partitions before domain weighting. Adaptive Query Execution is
enabled, but the actual file count and size are recorded because Spark may
coalesce or split work.

## Credential-free planning

```bash
python benchmarks/lakehouse_scale/benchmark.py \
  --profile sample \
  --mode plan \
  --manifest-out /tmp/hospitality-sample-plan.json

python benchmarks/lakehouse_scale/benchmark.py \
  --profile 10tb \
  --mode plan \
  --output-root abfss://benchmark@storage.dfs.core.windows.net/hospitality \
  --confirm-large-run YES_I_UNDERSTAND_COSTS \
  --manifest-out /tmp/hospitality-10tb-plan.json
```

Planning mode imports no PySpark module, contacts no cloud service, and needs no
credentials.

## Execute in Databricks or another managed Spark runtime

```bash
python benchmarks/lakehouse_scale/benchmark.py \
  --profile 100gb \
  --mode generate \
  --output-root /Volumes/hospitality_benchmark/scale/raw \
  --run-id 20260720-100gb-01 \
  --confirm-large-run YES_I_UNDERSTAND_COSTS
```

After tuning and validating 100 GB:

```bash
python benchmarks/lakehouse_scale/benchmark.py \
  --profile 1tb \
  --mode generate \
  --output-root /Volumes/hospitality_benchmark/scale/raw \
  --run-id 20260720-1tb-01 \
  --confirm-large-run YES_I_UNDERSTAND_COSTS \
  --full-reconciliation
```

The final 10 TB run should use the same command with `--profile 10tb` only after
the 1 TB run meets runtime, reliability, and cost gates.

## Credential rules

Never commit or print:

- Databricks personal access tokens
- Azure client secrets
- storage account keys or SAS tokens
- service-principal private keys
- cloud provider access keys
- `.databrickscfg`, `.env`, Terraform state, kubeconfig, or credential JSON

Use GitHub Actions OIDC/workload identity or a managed identity. Restrict access
to the benchmark catalog, storage prefix, and job. Add cost tags, cluster
policies, auto-termination, a maximum worker count, and a budget alert before
running a large profile.

The script accepts only storage locations and identifiers. It has no credential
arguments by design.

## Evidence policy

`evidence/run_manifest.template.json` has status
`PLANNED_NOT_EXECUTED`. Do not change that status manually. An executed
manifest must be generated by the workload and should include:

- source commit
- exact profile and output format
- target logical bytes
- actual compressed storage bytes
- estimated and optionally reconciled row counts
- file count by domain
- runtime by domain and total runtime
- anomaly rates
- workload limitations
- no secrets, workspace URLs, account IDs, or customer identifiers

A real report should also add sanitized Spark UI screenshots or exported metrics,
cluster shape, DBUs/cloud cost, spill, shuffle, skew, retry counts, and any
failed tuning attempts. Imperfect evidence with clear analysis is more credible
than a flawless unsupported claim.
