# Commit Message Standard

Use concise, imperative engineering messages that explain the change and its operational purpose.

Preferred examples:

- `Add point-in-time member feature contract`
- `Enforce seasonal-baseline forecast gate`
- `Harden Kubernetes readiness and disruption controls`
- `Document model rollback and replay procedures`

Avoid promotional or vague messages such as `final update`, `principal-level project`, or `misc changes`. Repository history should function as an engineering audit trail.
